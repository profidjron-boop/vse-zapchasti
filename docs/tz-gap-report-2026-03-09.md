# TZ Gap Report — 2026-03-09

Источник требований: `docs/requirements.md` (v1.1)  
Проверка по фактическому коду/роутам/экранам репозитория.  
Детализация по каждому пункту: `docs/tz-implementation-matrix-2026-03-09.md`.

## 1) Итог
- Статус: `strong pre-prod / release-candidate`.
- Обязательное ядро v1.1: реализовано.
- Оставшиеся задачи: в основном операционные и интеграционные (включение провайдеров/прод-инфры), плюс опциональные функции.

## 2) Реализовано (обязательные блоки v1.1)
- Публичные сценарии:
  - каталог/поиск/фильтры/карточка товара;
  - VIN-заявка (как заявка менеджеру);
  - сервис-заявка (как заявка менеджеру);
  - корзина guest + one-click + checkout;
  - история заказов по телефону;
  - избранное.
- API публичных write/read сценариев:
  - `POST /api/public/orders`, `GET /api/public/orders/history`;
  - `POST /api/public/service-requests`, `GET /api/public/service-catalog`;
  - `POST /api/public/vin-requests`.
- Админка по ТЗ:
  - товары/категории/цены;
  - VIN и service заявки со статусами;
  - заказы;
  - отчёты (минимально);
  - контент-разделы (о компании/контакты).
- Импорт и режимы обновления:
  - CSV/XLSX import pipeline;
  - режимы `manual | hourly | daily | event` через `import_products_update_mode`;
  - запуск через `scripts/import-products.sh`.
- Security/Ops baseline:
  - consent + audit fields по публичным формам;
  - rate-limit с Redis backend и in-memory fallback;
  - `health`/`ready`, `trace_id`/`X-Request-Id`, structured error envelope;
  - verify/runtime/release gates.

## 3) Частично / зависит от решения внедрения
- Предоплата записи:
  - техническая поддержка флага/суммы есть в service catalog;
  - полноценный платёжный контур (54-ФЗ, возвраты, провайдер) не включён по умолчанию в v1.1.
- Уведомления Email/SMS/мессенджер:
  - код каналов есть (`NOTIFY_*`, webhook allowlist);
  - финальное включение зависит от выбранных провайдеров/секретов на внедрении.
- 1С/ERP:
  - принят путь import-first;
  - online sync не включён в текущий scope (по ADR/tech design).

## 4) Необязательное (optional scope)
- Связка `Запчасть + установка` как отдельный продуктовый поток:
  - в текущем коде отдельного публичного flow не обнаружено;
  - по ТЗ этот блок опциональный.

## 5) Что доделать до “fully hardened prod”
1. Заполнить прод-операционные данные в handoff/state документах (домены, владельцы доступов, SLA/эскалации).
2. Зафиксировать финальную прод-конфигурацию провайдеров уведомлений (SMS/мессенджер/email) и секретов.
3. При необходимости включения предоплаты — запустить отдельный подпроект платёжной схемы (54-ФЗ/возвраты/провайдер) и только после этого включать фичу.
4. Поддерживать coverage и runtime/release gates на текущем уровне как обязательный pre-release check.
